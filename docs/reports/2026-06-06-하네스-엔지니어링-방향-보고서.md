## 2. AS-IS — 지금까지 어떻게 개발했나

### 2.1 개발 방식

**"대화형 바이브 코딩 → main 직행 → 운영에서 눈으로 확인"**

1. 사용자가 한국어로 요구사항/불편을 설명
2. AI 에이전트(Claude Code)가 코드 탐색 → 구현 → 커밋 → main 푸시
3. Vercel 자동 배포 (약 30초~1분)
4. 사용자가 실서비스에서 직접 확인 → 피드백 → 반복

| 지표 | 값 | 해석 |
|---|---|---|
| 총 커밋 | 247개 (2026-05-25 ~ 06-06) | — |
| 집중 구축 기간 | **이틀** (05-25: 200커밋, 05-26: 35커밋) | 사실상 2일 만에 제품 골격 완성 |

### 2.3 효과 평가 — 무엇이 좋아졌나

#### ① 개발 속도 (가장 큰 효과)

- 전통 방식 추정: 동일 범위(PWA + 관리자 + 인증 + AI 연동 + 인쇄 시스템 + 기기 간 동기화)를
  1인 개발 시 **4~8주** 수준의 작업량.
- 실제: **집중 2일 + 간헐 유지보수**. 속도 면에서 약 10~20배 효과.
- 요인: 빌드 없는 단순 스택 + AI 에이전트 + "운영 직행" 파이프라인의 결합.

#### ② 인프라 비용 — 사실상 0원 운영

| 항목 | 선택 | 월 비용 |
|---|---|---|
| 호스팅·함수 | Vercel Hobby | $0 |
| 데이터 저장 | ~~Vercel Blob~~ → **GitHub 커밋** (마이그레이션 완료) | $0 |
| 이미지·동기화 | Cloudflare R2 무료 한도(10GB) | $0 |
| AI 생성 | Gemini 무료 한도(일 1,500건) | $0 |
| 백업·감사 | git 이력 (별도 시스템 불필요) | $0 |

→ Blob 제거 마이그레이션이 핵심 의사결정. 비용 0 + 롤백(`git revert`) 공짜 확보.

#### ③ 성능

- 읽기 경로에 외부 API 호출 없음 (정적 파일 직접 로드) → 빠른 응답.
- Paged.js 자체 호스팅으로 외부 CDN 의존 제거 (오프라인·CSP 강화).
- 트레이드오프: 쓰기 반영 지연 30초~1분 (재배포 대기) — 개인용으로 수용 가능.

#### ④ 보안

- CSP·HSTS·COOP 등 보안 헤더 체계적 구성.
- innerHTML 회피(DOM 노드 조립) 등 XSS 의식적 방어 패턴 정착.

### 2.4 AS-IS의 빚 (리스크)

| # | 문제 | 영향 |
|---|---|---|
| 1 | **검증 루프 부재** — 테스트·린트·CI 0개 | 버그가 운영에서 발견됨. 회귀(regression) 무방비 |
| 2 | **컨텍스트 휘발** — CLAUDE.md/AGENTS.md 없음 | 매 세션 에이전트가 구조를 재탐색 (토큰·시간 낭비, 오판 위험) |
| 3 | **거대 단일 파일** — flash.js 2,636줄 | 수정 정확도 하락, 부작용 추적 곤란 |
| 4 | **데이터 무검증** — 카드 JSON 수천 줄이 스키마 검사 없이 커밋 | 데이터가 이 제품의 핵심 자산인데 가장 무방비 |
| 5 | **설계·의사결정 기록 부재** | "왜 이렇게 했더라"가 대화 로그에만 존재, 휘발됨 |
| 6 | 죽은 하네스 — settings.json이 존재하지 않는 `npm run typecheck` 허용 | 하네스 설정과 실제의 불일치 |

## 3. 하네스 엔지니어링 — 개념과 이 프로젝트의 해석

> 모델(AI)이 코드를 쓰는 시대에, 사람의 역할은 코드 한 줄이 아니라
> **에이전트가 스스로 맞았는지 알 수 있는 환경(harness)을 설계**하는 것.

4대 구성 요소와 본 프로젝트의 현재 좌표:

| 구성 요소 | 정의 | AS-IS | TO-BE |
|---|---|---|---|
| 컨텍스트 | 에이전트가 매번 다시 배우지 않게 하는 지식 파일 | 없음 | AGENTS.md + ARCHITECTURE.md + docs 체계 |
| 검증 루프 | 에이전트가 자기 결과물을 스스로 채점 | 없음 | 테스트 코드 + 데이터 검증기 + 스크린샷 증빙 |
| 가드레일 | 잘못된 변경이 운영에 닿기 전 차단 | 없음 | 훅(자동 구문검사) + CI 게이트 |
| 피드백·기록 | 결과와 결정이 다음 작업에 축적 | 대화 로그에만 | exec-plans·결정 기록·품질평가 문서화 |

## 4. TO-BE — 방향 수립

### 4.1 운영 원칙 (설계 가이드)

1. **반복 검증 의무화**: 모든 코드 변경은 `계획 → 작성 → 검증` 사이클을 강제하되,
   검증은 **자동화(훅·스크립트)** 로 수행해 추가 비용(시간·토큰)이 들지 않게 한다.
2. **증거 기반 완료 선언**: "고쳤습니다"가 아니라 —
   - 테스트 코드 작성 → 실행 → **통과 출력 제시**
   - UI 변경은 **스크린샷 제시** → 사용자 확인 후 배포
3. **기록은 작업의 부산물로**: 별도 문서화 시간을 들이지 않고,
   계획서(exec-plan)·결정 로그가 대화 중 자동으로 docs에 쌓이게 한다.
4. **개발 환경을 하네스에 내장**: 로컬 실행법·시크릿 주입·검증 명령을
   AGENTS.md에 기록해 어떤 세션·어떤 에이전트든 즉시 재현 가능하게 한다.
5. **멀티에이전트 분업**: 조사(Explore) / 설계(Plan) / 구현(메인) / 리뷰(code-review)를
   역할별 에이전트로 분리해 상호 검증한다.

### 4.2 목표 디렉터리 구조

```
itpe/
├─ AGENTS.md                  ← 에이전트 진입점: 규칙·명령어·함정·환경설정 (CLAUDE.md가 참조)
├─ ARCHITECTURE.md            ← 시스템 구조: 데이터 흐름, API 지도, 저장 모델
├─ SECURITY.md                ← 보안규칙: CSP 정책, 인증 모델, 비밀정보 취급, XSS 패턴
├─ docs/
│  ├─ design/                 ← (기존) 디자인 시안
│  ├─ product-specs/          ← 기능 명세: 찾기, 인쇄, TTS, 체크 동기화 등 기능별 1문서
│  ├─ exec-plans/             ← 실행 계획서: 작업 전 계획 수립 → 완료 후 결과 추기
│  ├─ references/             ← 외부 참조: Vercel/R2/Gemini 설정, 원본 엑셀 등
│  ├─ reports/                ← 본 보고서 등 평가·회고
│  └─ QUALITY.md              ← 품질평가 기준표 + 회차별 평가 기록
├─ tests/
│  ├─ data.test.mjs           ← 카드 JSON 스키마·중복·무결성 검증
│  ├─ api.test.mjs            ← API 단위 테스트 (인증, cards CRUD)
│  └─ e2e/                    ← Playwright: 핵심 3흐름 + 스크린샷 증빙
├─ scripts/
│  └─ check.mjs               ← 통합 검증: js 구문 + 데이터 스키마 (커밋 전 필수)
├─ .claude/settings.json      ← 훅: JS 수정 시 자동 구문검사, 권한 정비
└─ .github/workflows/ci.yml   ← main 푸시 시 check + test 자동 실행
```

### 4.3 검증 사이클의 비용-제로 설계

핵심: **검증을 사람이 하면 비용이고, 하네스가 하면 공짜다.**

| 단계 | 도구 | 비용 |
|---|---|---|
| 코드 수정 직후 | Claude Code **PostToolUse 훅** → `node --check` 자동 실행 | 0 (로컬, 수 ms) |
| 커밋 전 | `npm run check` (구문 + 데이터 스키마) | 0 (로컬, 수 초) |
| 푸시 후 | GitHub Actions CI (무료 한도) | 0 |
| UI 변경 시 | Playwright 스크린샷 → 대화에 제시 | 로컬 실행, 토큰 소량 |
| 컨텍스트 | AGENTS.md 선탑재 → 세션마다 재탐색 제거 | **토큰 절감** (순이익) |

> 토큰 관점: AGENTS.md(약 2~3천 토큰)를 매 세션 싣는 비용 <
> 매 세션 구조 재탐색(오늘 기준 도구 호출 6회 이상, 수만 토큰)의 비용.
> 컨텍스트 문서는 비용이 아니라 **절감 장치**다.

### 4.4 멀티에이전트 운영 모델

```
사용자 요구
   │
   ├─ [Explore 에이전트] 코드/데이터 조사 (읽기 전용, 병렬)
   ├─ [Plan 에이전트] 설계안 작성 → docs/exec-plans/ 에 기록
   │
   ▼
[메인 에이전트] 구현 ── 훅이 즉시 구문검사 (자동)
   │
   ├─ tests/ 실행 → 통과 증거 제시
   ├─ UI 변경 시 Playwright 스크린샷 제시
   │
   ▼
[code-review 에이전트] 커밋 전 diff 리뷰 (/code-review)
   │
   ▼
main 푸시 → [CI] 최종 게이트 → Vercel 배포
   │
   └─ 결과를 exec-plan 에 추기 (완료 기록)
```

### 4.5 단계별 실행 계획

| Phase | 작업 | 규모 | 기대 효과 |
|---|---|---|---|
| **P1. 컨텍스트** | AGENTS.md, ARCHITECTURE.md, SECURITY.md 작성. docs/ 체계(product-specs·exec-plans·references·reports) 신설. settings.json 정비 | 반나절 | 세션 간 지식 영속화. 토큰 절감. 어떤 에이전트든 즉시 투입 가능 |
| **P2. 검증 루프** | scripts/check.mjs (js구문+카드 스키마), tests/data.test.mjs, npm scripts, PostToolUse 훅 | 반나절 | 데이터 7,400줄 변경도 안심 커밋. 에이전트 자가 채점 |
| **P3. 가드레일** | GitHub Actions CI, /code-review 루틴화, QUALITY.md 기준표 | 2~3시간 | 운영 직행의 최후 방어선. 품질의 측정 가능화 |
| **P4. UI 증빙** | Playwright 도입: 로그인→단원 진입→찾기 점프 3흐름 + 스크린샷 자동 캡처 | 반나절 | "고쳤다"가 아니라 화면 증거로 확인. 회귀 자동 감지 |
| **P5. 구조 개선** | flash.js 모듈 분할(찾기/TTS/인쇄/타이머), JSDoc + tsc --checkJs | 1일 | 이후 모든 수정의 정확도·속도 상승 |

P1→P2가 최우선. P4(스크린샷)는 사용자 확인 문화에 직결되므로 P3과 병행 권장.

### 4.6 지키는 것 (하지 않을 일)

- **빌드 도구·프레임워크 도입 안 함** — 바닐라+무빌드의 단순함이 이 프로젝트 속도의 원천.
- **브랜치 전략 복잡화 안 함** — 일상 수정은 main 직행 유지. 큰 변경만 Preview 배포 활용.
- **문서를 위한 문서 안 만듦** — exec-plan·결정 기록은 작업 과정의 부산물로만 생성.
- **유료 인프라 도입 안 함** — 현 0원 구조 유지가 제약 조건.

## 5. 결론

ITPE Flash는 "단순 스택 + AI 에이전트 + 운영 직행"의 조합으로 **2일 구축·0원 운영**이라는
뛰어난 속도·비용 효율을 달성했다. 다만 그 속도는 검증과 기록이라는 빚 위에 서 있다.

다음 단계는 속도를 죽이는 절차 추가가 아니라, **에이전트가 스스로 채점하고(검증 루프),
스스로 기억하고(컨텍스트·기록), 화면 증거로 보고하는(스크린샷) 하네스**를 까는 것이다.
P1~P2(컨텍스트+검증)만으로도 현재 리스크의 대부분이 해소되며, 추가 비용은
로컬 스크립트와 무료 CI 한도 내에서 0에 수렴한다.
